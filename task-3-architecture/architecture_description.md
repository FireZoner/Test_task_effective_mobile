Описание компонентов диаграммы:
1) Уведомления могут быть получены от трёх источников - мобильное приложение (например, при смене пароля пользователем), событие от бэкенда (например, изменение статуса заказа) и через админ-панель (например, запуск рекламной кампании маркетологом).
2) Все уведомления попадают на API Gateway, где проходят первичную валидацию (например, по JSON-схеме). Тут же происходит проверка, кто отправил запрос, и перенаправление в нужный сервис (в нашем случае это только лишь сервис уведомлений). Также API Gateway на этом этапе может стать буфером, если случаются ситуации перегрузок уведомлениями.
3) Сервис уведомлений получает уведомления, проверяет по БД, не отписался ли пользователь от рассылок, из того же БД берёт шаблон уведомления и подставляет в него данные, полученные от предыдущего компонента. Затем кладёт задачу в брокера.
4) Брокер хранит уведомления. Предотвращает потерю и сглаживает пики нагрузок.
5) Сервис отправки получает уведомления из брокера, смотрит, кому нужно их отправить, идёт в БД/кэш (например, Redis), где хранится адрес пользователя/токен и тип устройства и перенаправляет в соответствующий провайдер.
6) Провайдеры - это внешние сервисы. Сервис отправки отдаёт им команду, а они уже сами доставляют сообщение.
